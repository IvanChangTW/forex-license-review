SECTION 01

關鍵術語

本文件大量使用以下技術用語、先在此一次定義、後續直接引用：

核心詞彙表

AGENTS.md / CLAUDE.md
AI agent 開新對話時自動載入的「系統提示主檔」。AGENTS.md 是 Codex 用的、CLAUDE.md 是 Claude Code 用的。內容是工作背景 + 紀律 + 記憶索引。本架構的 AGENTS.md 由 sync 腳本動態編譯產生、非手寫。
SSOT（Single Source of Truth）
同一個事實只在一處權威保存、其他地方都是衍生副本。本架構選 claude-memory GitHub repo 為 SSOT — 所有 memory 真實內容只在這、其他地方（AGENTS.md、agent 本機 dir）都是 sync 出來的。
Scope
Memory 檔的隔離維度、決定哪些檔在哪些機器上載入。本架構分三層：shared/（四端共用）、office/（公司 Windows only）、home/（家裡 Mac only）。
codex_tier
Memory 檔 frontmatter 標籤、控制檔在 AGENTS.md 編譯時的呈現粒度。四級：core（全文進）、summary（只進摘要段）、detail（只列檔名、agent 需要再讀）、off（不進）。
Tier compilation（編譯版 AGENTS）
5/11 引入的機制 — sync 腳本不再直接拼接所有 memory 檔、而是依 codex_tier 標籤分層編譯。把 AGENTS.md 從 ~32 KiB 壓到 ~13.6 KiB、prompt 預算打開。
SessionStart hook
Claude Code 新對話開始時自動執行的腳本。本架構用於：git pull 拉最新 memory、跑 sync 腳本、跑 health check。每天每端各跑一次、不需人工觸發。
Sync 腳本
四端各自的同步工具：sync-office.ps1、sync-home.sh、sync-codex.{ps1,sh}。職責：git pull、編譯 AGENTS.md、跑 health check。
Cross-agent baton
跨 AI agent 工作交接的紀律。當工作從一個 agent（如公司 Claude Code）接到另一個（如家裡 Codex），用「baton」隱喻接力棒、避免 context 在切換時遺失。
Selective load / includes directive
5/13 引入的 frontmatter 機制。檔可宣告 includes: [其他檔] 形成依賴圖、配合 codex_load: include_only flag 讓被引用檔預設不載、只在 parent include 時才出現。目的是減少 AGENTS.md prompt 佔用。
Inline boundedness
在 memory 檔內 inline 宣告「自己什麼時候該被拆」+「條目什麼時候該被移走」的紀律。例如：backlog 檔自己寫「>10 條時拆出 BACKLOG.md」。避免規則寫在外部然後被忘記。
memory_health / lifecycle_health
兩個 enforcement 腳本（不只是 reference 文件）— 跑起來會偵測 AGENTS.md 接近上限、cross-scope leak、檔長期沒更新等問題、給 warning。從 5/12 開始把紀律從「文件描述」升級到「執行檢查」。
MCP（Model Context Protocol）
Anthropic 推的 AI agent 接外部工具的標準協議。本架構用於 Claude Chat 接 Notion / Google Drive / Calendar 等服務。

整體軸與時間表

9 天內、從零建立到生產級的個人 AI 工作流記憶治理系統。每一天加的不是 incremental feature、而是 architectural step。下圖顯示 9 個 milestone：

關鍵特徵

  • 每階段都先研究外部對照後再決定內部設計（cc-switch / Hermes / PLUR / superpowers / Cowork）— 不閉門造車
  • 每階段產出至少一條可重用方法論、且方法論寫進 memory 自己（feedback_*.md）
  • 每階段都經得起跨 4 端 + Notion 工作日誌的證據追溯
  • 設計哲學一以貫之：explicit authoring、git 為 SSOT、local-first、與 31 年手寫日記哲學對齊

核心架構圖

為什麼 git 為 SSOT 比 cloud-backed memory 穩

可 diff、可 rollback、可離線：git 提供完整歷史、出錯可回滾、網路斷線仍能 work。

無 vendor lock-in：Anthropic 哪天倒了、memory 仍是純 markdown、平台中立。

跟 31 年手寫日記哲學一致：explicit、可追溯、可遷移、人類可讀。

SECTION 03

逐日演化

Day 0 · 05/06Notion 工作日誌資料庫誕生

當天還沒有「memory infra」概念。三件事打下後續基礎：RunLossReport AHK 重構 + Notion MCP 接通、family-assets 從零打造 + 公網部署、family-allowance 主題系統大擴增。沒有意識到「memory 是獨立 problem」、只是把日常結論寫進 Notion 試試。

Day 1 · 05/07跨機器同步架構奠基

跨出 Notion 後第一個真正的 memory infra 設計。claude-memory repo 落定：shared/ + office/ + home/ + scripts/。當天落定的 5 個關鍵設計：

設計為什麼這樣設
GitHub repo 為 SSOT與 Claude Code「本地檔 + auto-load」機制對齊、不把 Notion 塞進不擅長的 use case
MEMORY.<scope>.md 由 sync 串接生成、不 commit索引片段各 scope 自己維護、避免兩地手動同步
reference_claude_memory_repo.md（meta-memory）「memory 自己的規則」也是 memory；新對話 auto-load 就懂規則、不用每次重新解釋
SessionStart hook 自動 git pull + hash 比對 no-op解掉「100 個無意義 .bak」+「PowerShell Set-Content 偷加 CRLF」兩個隱形坑
.gitattributes 鎖死 line endings.sh 在 Mac 不會炸 — 經驗值不是試出來的

這天之後的所有改動、都是這個架構的延伸或精修。

Day 2 · 05/08Codex 跨機器整合

把 Codex（公司 + 家裡兩個 host）也接進 memory 系統。sync-home.sh 落地、Codex 跨機器 memory 整合 + 接 Notion + 抽 notion_helper。四端架構正式成形。但 SOP 對稱仍有 debt、5/09 收。

Day 3 · 05/09Codex SOP 對稱化 + 跨對話視窗

「四端可用、但 SOP 不對稱」的 debt 收尾。「跨 agent baton」概念雛形浮現 — 但還沒命名、是隱含的開工 SOP。

Day 4 · 05/10紀律驗證 + 方法論首次正式提取

family-allowance 一日 22 commits / 10 件大改、過程中 SOP 被三度驗證。當天提取首批正式方法論（第 8/9 條 firm rule、同日成功應用）。「方法論文件化」這個 pattern 第一次自然發生 — 不是刻意做、是工作自然產出。後續 inline_size_limits 等都是這個 pattern 的延續。

Day 5 · 05/112.1 升級：tier 編譯 + cross-agent baton 閉環

從拼接到編譯

AGENTS.md 一路加東西、~32 KiB 接近 Codex prompt 預算上限。引入 codex_tier 四級標籤（core/summary/detail/off）、把 AGENTS.md 從直接拼接改成依 tier 編譯。

為什麼 tier 編譯比 flat 拼接穩

容量可控：core 嚴選必載、summary 提供索引、detail 按需讀取。AGENTS.md 不會無限長大。

同份檔多種解析度：Claude（看全文）和 Codex（看編譯版）能用同一份 source、不需要 fork 兩套 memory。

新增 SOP 不撞天花板：把詳細內容降為 detail tier、AGENTS 主檔保持精簡、後續演化空間打開。

Cross-agent baton 概念閉環

當天首次正式運作「Claude Code 提疑問 → 使用者橋接 → Codex 驗證 → 回寫」的 round-trip、約 12 分鐘。寫進 reference_codex_memory_strategy.md、首次成為可命名的 architectural pattern。但此時還是口頭傳遞為主、markdown 為輔（後續會 invert）。

Day 6 · 05/12從紀律到自動化

5/11 之前所有「紀律」都是 markdown 描述（agent 讀過自己記得）。5/12 開始把紀律寫成 enforcement script。

Step內容本質變化
memory_health.ps1 落地AGENTS 接近上限警告 / core 漏掉偵測 / cross-scope leak / sync timestamp紀律 → 哨兵
project_lifecycle_health.ps130/90 天 commit-age 自動掃描、warn / suggest archive原則 → 執行
Inline 邊界全套 propagate8 份 status card + MEMORY.shared + feedback_daily_workflow 都加邊界外部規則 → inline 自我宣告
feedback_daily_workflow.md 拆檔hub + morning / evening / memory_write / cross-day / home-Claude 5 個專檔monolith → 模組化
Detail tier 雙層驗證Claude 端 sync-codex 加 5 行存在性檢查 + Codex 端 SOP first-5-message verify單點假設 → 雙層 fallback
新方法論：Inline boundedness 兩層 pattern

條目層：什麼時機把條目從檔內移走（例：P0 done → 移除、不留 ✅）

檔層：什麼時機本檔該拆成多檔（例：backlog >10 條 → 拆 BACKLOG.md）

且閾值要可觀察（>10 條 / >30 天無 commit ✓；「感覺太肥」✗）。寫進 feedback_inline_size_limits.md、5/14 selective load 決策時自然應用。

為什麼 inline 邊界比外部 policy 穩

自我宣告、不會被忘記：規則寫在檔內、編輯時就看到、外部 policy 容易隨時間漂移失憶。

觸發條件強制可觀察：用具體數字（>10 條）取代主觀感受（「太肥了」），避免 reviewer 之間 disagree。

方法論可重用：同 pattern 用在 backlog、status card、Notion 日誌、AGENTS.md — 不必為每個容器設計新規則。

cc-switch 研究（5/12 evening）

研究 cc-switch repo、原本標題寫「吸收 CC-Switch guardrails」、事後驗證 cc-switch 跟你的場景不對應（它做 provider config switching、你做 memory governance）。但「guardrail 該被產品化」這個設計衝動是真的、驅動了當天從紀律到自動化的轉折。研究外部對象的價值不只在「採納什麼」、也在「觸發什麼設計衝動」。

Hermes 提案 → Codex 實作（同日）

Claude Chat 起草 9 章節 markdown 提案 → Codex 接到後直接落地 Proposal 2/3 + Proposal 1 Path A pilot。Codex 補上提案漏掉的 codex_load: include_only flag（提案只寫 includes: 陣列、沒寫如何讓被引用檔預設不進選集）。

家裡端 reliability round（同日晚）

家裡 Codex 補上 memory_health.sh（Bash parity）、清掉 7 個 shared cross-scope warning、macOS shasum locale warning 局部修法。

新方法論：Pilot → Reliability baseline → Expansion

當 pilot 證明 breadth 夠、下一步應補 reliability baseline、不是繼續擴 coverage。很多 project 死在「pilot 跑通了就直接擴大」、跳過 reliability baseline 階段、結果擴大後 reliability debt 一次性爆出來。

5/13 家裡 Codex 沒去加更多 includes、而是先補 Bash parity 跟 health check — 是這條方法論的活例。

為什麼「修問題、不修 check」比反向穩

Health check 抓到 warning 時、預設動作是「驗證 warning 是否真實」、不是「降 check 標準」。

5/13 抓到 7 個 cross-scope warning、選擇修 shared 檔內容、不選擇調低 check 敏感度。反向路徑會把 health check 變成 placebo、5/12 從紀律到自動化的成果反向退步。

Day 8 · 05/14SOP 正式化 + parity 收斂

5/13 累積的觀察、5/14 收斂成正式 SOP。三件 architectural step：

(a) Cross-agent baton SOP 正式化

新增 shared/feedback_cross_agent_baton.md。雙層規則：

層角色規則
PrimaryMarkdown 留存層Source-of-truth、必信
Secondary口頭交接補速度、不替代 markdown

順序顛倒就 drift。Baton 規則 propagate 進 4 個入口（MEMORY.shared / Claude 早上開工 / 家裡 Claude SOP / Codex 日常 workflow）。

為什麼 markdown-primary 比 verbal-primary 穩

持久性：markdown 寫進 repo、跨日跨 session 仍在；口頭內容隨 session 關閉消失。

可追溯：markdown 有 git history、可追究決策來源；口頭內容無 audit trail。

跨 agent 對等：markdown 對任何 agent 都能 read；口頭傳遞需要人為轉述、有 distortion 風險。

速度由 secondary 補：口頭依舊有用、但是「速度補充」不是「source of truth」。

(b) memory_health 雙端 defaults 共用

新增 scripts/memory_health.common.env、抽出 6 個最容易 silent drift 的 defaults：MAX_BYTES、WARN_BYTES、DEFAULT_STALE_HOURS、REQUIRED_CORE_ENTRY、VALID_TIERS、VALID_LOADS。

沒抽的部分（檢查邏輯、輸出格式、subprocess handling）保留兩份實作 — Bash vs PowerShell 差異太大、強求同碼會引入更多 bug。

為什麼「只抽 defaults」比「抽全部 code」穩

跨 shell parity 不是追求同碼、是先共用最容易 drift 處。Bash 跟 PowerShell 在 subprocess handling、locale、error code 慣例上差異大、強求同碼會引入更多 bug。

共用最穩定的層（defaults / thresholds）、保留兩 shell 各自做最熟悉的事（檢查 logic）— 是真正的 platform-aware unification。

(c) Selective load 第 3 題：先 observe

評估 Path A 擴張到 project_status_* 的時機、結論「先 observe 不擴」。觸發條件設明確：AGENTS 逼近 26 KiB / 依賴關係重複出現。這是 feedback_inline_size_limits.md 兩層 pattern 在生產環境第一次自然應用 — 5/11 寫下規則、5/14 在不被提醒情況下用它做決策。

SECTION 04

三個 abstraction layer

整個演化可以拆三層、且每層各自有 evolution arc。三層之間有自指（self-referential）模式 — Layer 1 的 pattern 被觀察、寫成 Layer 3 的 methodology、再被 Layer 2 的 enforcement script 實作。

自指循環的具體例子

觀察起點（Layer 1）方法論（Layer 3）執行碼（Layer 2）

SECTION 05

Cross-agent baton 演化軌跡

整段歷程裡「跨 agent 協作」這條軸演化特別清晰、值得單獨 visualize：

為什麼雙層 baton 規則比單層穩

單一 channel 都有失效模式：純口頭忘記、純 markdown 太慢。雙層讓兩種 channel 各補對方的不足。

明確 primary/secondary 避免歧義：如果兩種 channel 內容衝突、規則明確以 markdown 為準、不會卡在「誰對」。

順序顛倒就 drift：5/11 的口頭主導其實已踩坑（提案內容太詳盡、跳過 ask-back 環節）— 反向順序設定才是 SOP 正式化的關鍵。

SECTION 06

AGENTS.md bytes 容量趨勢

AGENTS.md（Codex 用的 system prompt 主檔）容量是容量壓力的 single most important indicator。5/11 編譯機制把它壓到 ~13.6 KiB、但 9 天累積 architectural change 之後已逼近警戒線：

+38%
3 天淨增（5/12→5/14）
74.5%
當前佔預算
1.6 KiB
距警戒線
~3-5d
同速度逼近警戒線

「先大後小 transition 期」這個 5/13 寫進策略檔的觀察、到 5/14 仍只看到「先大」、沒看到「後小」。三個合理解釋：

  1. 反覆 architectural change 期、每次都加 SOP / methodology 條目進 shared、量本來就會持續成長
  2. Selective load 還沒大規模 rollout、include_only 檔少、reduction 效果未顯
  3. 真的就需要這些 bytes、74.5% 仍在預算內、不痛

26 KiB warning line 逼近時、應該重審「先 observe」決策、考慮把 selective load 擴張到 project_status_*。

SECTION 10

設計哲學

整個架構走下來、underlying principles 可以總結為 7 條。每條都附「為什麼這樣比反向選擇穩」的說明：

原則內容 + 為什麼穩
1. Explicit authoring over auto-emerge Memory 由人 curate、不由 LLM 自動 derive。
為什麼穩：完整 control、git 可 audit、跟 31 年手寫日記哲學一致、不依賴 LLM 黑箱判斷。
2. Git as single source of truth 所有 memory 變動可 diff / rollback / git log。
為什麼穩：版本控制完整、distributed、離線可用、無 vendor lock-in。
3. Local-first Memory 在本機檔系統、不依賴外部 API endpoint。
為什麼穩：隱私、不受 vendor 服務中斷影響、不受網路延遲影響、不受 API 收費政策影響。
4. Platform-agnostic data portability 用 markdown、不用 vendor-specific format。
為什麼穩：Anthropic 哪天倒了、memory 仍可讀。對應 31 年日記哲學的延伸。
5. Filesystem as unification layer Claude Code / Codex 都認 filesystem、可被 git 串起來。
為什麼穩：unified layer 必須是兩個 agent 都認的、filesystem 是當前最低共識基礎。Cowork 不認 filesystem → 不收編。
6. Methodology written into memory itself 規則寫進 memory 自己（feedback_*.md），下次 session auto-load 就懂。
為什麼穩：規則不外掛、不靠人記、不會被忘記、跨 session 自動 propagate。
7. Minimum blast radius for changes _v2 不覆寫、局部修不全域改、common.env 只抽 defaults 不抽 logic。
為什麼穩：可回退、可隔離 bug、可驗證無副作用、降低任一改動的風險面。
這 7 條跟 Anthropic 自家產品演化方向部份對立。但對立不是 bug、是設計權衡。
Anthropic 方向Ivan 方向
多 surface（Chat / Cowork / Code / Dispatch）Filesystem-unified 4 端、收斂 surface
Server-side memory derive（auto-emerge）Explicit markdown authoring
Cloud-backed memory providerLocal-first git repo
Vendor-managed conveniencePlatform-agnostic 31 年可追溯

Ivan 的路徑 trade off 了「自動化便利」、換來「31 年可追溯 + 完整 control + platform 中立」。

SECTION 12

結論

9 天從零到生產級。每天一個 architectural step、沒有空轉、沒有 cleanup-only 的日子。

最值得記住的三條觀察

觀察 1

Compound effect 真實存在。5/06 沒有 memory infra 概念、5/14 已經有正式 baton SOP + cross-shell parity + inline boundedness pattern 在生產應用。每天一個 step、9 步成林。如果中間某天 skip、後面整個鏈條會斷。

觀察 2

研究外部 + 設計內部 + 落地實作 + 提取方法論，這條 loop 是這幾天唯一的工作流。每外部研究都產出 internal 設計、每設計都進落地、每落地都產出方法論、方法論寫進 memory 自己。這條 loop 自我增強。

觀察 3

Memory infra 是 surface 收斂的對應物。Cowork 多 surface vs Ivan filesystem-unified — 兩條路都有 logic、但選了後者後、整個 memory infra 才能 work。SSOT 跟 surface 收斂是同一個哲學的兩面。
